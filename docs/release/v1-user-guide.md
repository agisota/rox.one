# ROX.ONE v1.0.0 — User Guide

Audience: ROX.ONE operators — anyone running the app, connecting agents to real
work, and shipping outcomes.
Scope: desktop app, headless server, web client, CLI. Single-user by default;
multi-tenant when an operator enables it.

This guide assumes you have already installed ROX.ONE. If you haven't, see
`README.md` → Installation. Screenshot placeholders are flagged with
`![Screenshot: …]` for the design pass.

---

## 1. Quick start

1. **Install.** Run the one-line installer from `README.md` or build from source.
2. **First launch.** Open ROX.ONE. The app creates `~/.rox/` and a default
   workspace.
3. **Choose an API connection.** Pick one of:
   - **Anthropic** (API key, or sign in with your Claude Max / Claude Pro
     subscription)
   - **Google AI Studio** (API key)
   - **ChatGPT Plus / Pro** (Codex OAuth)
   - **GitHub Copilot** (device-code OAuth)
   - Any **OpenAI-compatible** endpoint (OpenRouter, Vercel AI Gateway,
     Ollama, self-hosted)
4. **Start your first session.** Cmd+N. Type a prompt. Hit Enter.

That is the whole loop. Everything else is depth on top of this loop.

selected, Composer empty.]

## 2. Workspaces and sessions

A **workspace** is a folder of related work — a project, a client, a side
investigation. A **session** is a single agent conversation inside a workspace.

### Create a workspace

- Sidebar → `+ New workspace`.
- Give it a name. Optionally point it at a local directory (the agent gains
  filesystem access scoped to that path).

### Switch workspaces

- Click the workspace name in the sidebar header to open the picker.
- The active workspace controls which sources, skills, statuses, themes, and
  RBAC scope apply to new sessions.

### Delete a workspace

- Workspace picker → hover → `…` → Delete.
- Sessions inside the workspace are archived to `~/.rox/workspaces/<id>/` on
  disk; deletion is reversible for 30 days.

### Multi-session inbox

The session list is your inbox:

| Column | Meaning |
|---|---|
| Status | Todo, In Progress, Needs Review, Done — or any custom status you've defined |
| Flag | Star/unstar important sessions |
| Title | AI-generated by default; click to rename manually |
| Last activity | Most recent message timestamp |

**Keyboard:** `Cmd+1/2/3` to jump between sidebar / list / chat focus.
**Filtering:** type in the search bar to filter by title, status, label, or
content.

![Screenshot: Session inbox with mixed Todo / In Progress sessions and a
flagged one at the top.]

## 3. Composer modes

The Composer is where you start work. ROX.ONE exposes three **Mission Modes**
(`command`, `game`, `arena`), matching the Experience Layer in
`packages/shared/src/workbench/experience-layer.ts`:

| Mode | When to use | What changes |
|---|---|---|
| **Command** | Direct task execution. Most users live here. | Linear prompt → agent response, full tool access subject to your permission mode. |
| **Game** | Long-running missions with checkpoints, gates, and progression. | Adds verification gates, checkpoint cadence, and Quality / Readiness scoring against the mission objective. |
| **Arena** | Multi-agent swarm — several agents working in parallel against the same brief. | Adds dedupe, signal ranking, and a review board before any artifact lands. |

Switch modes from the Composer mode selector. The mode is per-session — you
can have a Game-mode mission running next to an Arena swarm and a Command
session for quick lookups.

![Screenshot: Composer with Mission Mode selector open, "Command" selected.]

## 4. Skills and sources

Skills and sources are how you teach ROX.ONE about your world.

### Add a skill

A **skill** is a per-workspace bundle of agent instructions, tool affordances,
and trigger patterns. They live at `~/.rox/workspaces/<id>/skills/`.

1. Tell the agent: *"Create a skill that does X. Trigger on phrases like Y."*
2. The agent drafts the skill file, asks for confirmation, and saves it.
3. Mention the skill mid-conversation with `@skill-name`.

No restart needed — skills become available the moment they save.

### Paste an MCP config JSON

You already have an MCP server config from another tool? Paste it. The agent
parses it, asks for any missing credentials, and installs it as a workspace
source.

```json
{
  "mcpServers": {
    "linear": {
      "command": "npx",
      "args": ["-y", "@linear/mcp-server"],
      "env": { "LINEAR_API_KEY": "${LINEAR_API_KEY}" }
    }
  }
}
```

### Connect to Linear, Gmail, Slack

The friendly path: *"Add Linear as a source."* The agent finds the public
API + MCP options, walks you through OAuth or API-key entry, and configures
everything. No config files, no setup wizards.

Supported source types:

| Type | Examples |
|---|---|
| **MCP servers** | ROX, Linear, GitHub, Notion, custom stdio or HTTP servers |
| **REST APIs** | Google (Gmail, Calendar, Drive, YouTube, Search Console), Slack, Microsoft 365 |
| **Local filesystems** | Any folder, Obsidian vault, Git repo |

OAuth credentials are stored encrypted with AES-256-GCM in
`~/.rox/credentials.enc`. They never reach an MCP subprocess unless you
explicitly pass them via the source's `env` field.

## 5. Permissions

ROX.ONE uses a three-mode permission system. Use **SHIFT+TAB** in the chat to
cycle between them per-session.

| Display | Internal | Behavior |
|---|---|---|
| **Explore** | `safe` | Read-only. All write/exec tools blocked. |
| **Ask to Edit** | `ask` | Prompts for approval before each write. Default. |
| **Auto** | `allow-all` | Auto-approves everything. Use with care. |

### Custom rules

Workspace → Settings → Permissions lets you write **Permission Rules** that
override the mode for specific tools, paths, or commands.

```jsonc
{
  "tool": "bash",
  "match": "git status*",
  "decision": "allow"     // always allow read-only git probes
}
```

Rules apply in order. The first match wins. Mode falls through if no rule
matches.

### The ROX.ONE Skill model

A skill can declare its own permission contract. The skill's contract is
**additive** — it can only narrow the active permission mode, never widen it.
If your session is in Explore mode and a skill asks for `bash` write access,
the request fails closed.

## 6. Long-running missions

Game mode unlocks missions that run for hours or days with periodic
checkpoints.

### Start a 6h / 24h / 72h mission

1. Switch the Composer to **Game** mode.
2. Click *New mission*.
3. Fill in:
   - **Title** — what is this mission?
   - **Objective** — the verifiable outcome
   - **Duration** — 6h, 24h, 72h, or custom
   - **Cadence** — how often to checkpoint (default: hourly for ≤24h, every
     4h for longer)
   - **Budget** — token cap, storage cap, agent count, VDI ceiling
4. Hit **Launch**.

### Monitor progress

The Mission Center surfaces:

- Live checkpoint stream
- Active validation gates (passing / failing / blocked)
- Artifacts produced so far
- Budget burn vs. cap
- Audit feed for every privileged tool call

![Screenshot: Mission Center showing a 24h mission at the 6-hour mark, three
green gates, one warning.]

### View checkpoints

Click any checkpoint to see:

- The agent's reasoning at that step
- The artifact diff (file changes, generated documents, tool outputs)
- The validation outcome
- A "Rewind to here" button — branch a new session from this checkpoint

### Important rule

A mission only **completes** when its final artifact exists AND all blocking
gates pass. Buying more capacity cannot complete a mission. Quality is
evidence-bound.

## 7. Public sharing

Generate a shortlink to share a session with someone who doesn't have
ROX.ONE installed.

### Generate a shortlink

- Session header → `Share` → `Create public link`.
- Choose an expiry: 24 hours, 7 days, 30 days, or "until revoked".
- Click **Create**. The shortlink copies to your clipboard.

### What gets sent

The shared bundle contains:

- All user messages
- All agent responses (text + structured content)
- Tool calls and their results, redacted of secrets / file contents the
  agent fetched from your machine
- Adaptive media (rendered tables, diagrams) — best-effort: some adaptive
  surfaces may not render in the public viewer (see Known Limitations)

### What is preserved on your machine

Everything. Sharing never deletes or moves the source session — your local
copy is the source of truth.

### Revoking

Session header → `Share` → `Revoke link`. The shortlink immediately stops
resolving; recipients see a "link revoked" page. Revocation is irreversible —
you'd have to generate a new link.

![Screenshot: Share dialog with expiry dropdown showing "7 days", "Until
revoked", etc.]

## 8. Themes and personalization

ROX.ONE has a **cascading theme model**:

```text
app theme (global, ~/.rox/theme.json)
  └── workspace theme (~/.rox/workspaces/<id>/theme.json, optional)
        └── session overrides (rare; per-session UI tweaks)
```

A workspace theme overrides only the fields it sets — unset fields fall back
to the app theme. This lets you have one global look-and-feel and per-client
accents per workspace.

### Change the app theme

Settings → Appearance → Theme. Pick from presets or load a JSON theme file.

### Change a workspace theme

Workspace settings → Theme → "Override app theme". Saves to that workspace's
`theme.json`.

### Build a custom theme

Tell the agent: *"Make a high-contrast dark theme for this workspace with a
purple accent."* It writes the theme file, you review the diff, and accept.

## 9. Settings → Team & permissions (RBAC)

Available when you're running in multi-tenant mode (`ROX_MULTI_TENANT=1` on
the server). Single-user installs see a simplified version.

### Invite a teammate

1. Settings → Team & permissions → `Invite`
2. Enter their email + initial role
3. They receive an invitation link; clicking it pairs their ROX ID to this
   workspace.

### Grant a role

Roles are defined per RBAC schema. A grant has:

- **Subject** — the user being granted the role
- **Role** — what they can do (`viewer`, `editor`, `admin`, or a custom role
  you've defined)
- **Scope** — where the grant applies: a specific `workspace`, an `org`
  (post-release), or `global` (server-wide)

Roles are additive — a user's effective permissions are the union of all
their grants.

### Revoke a grant

Team & permissions → click the user → click the grant → `Revoke`. Revocation
propagates immediately; subsequent requests from that user are denied.

visible.]

## 10. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| App won't open on macOS first launch | Gatekeeper quarantine on a freshly downloaded build | `xattr -d com.apple.quarantine "/Applications/ROX Agents.app"` (one-time per install) |
| "Connection refused" on remote server | Server not bound on `0.0.0.0`, firewall blocking the port, or wrong protocol (`ws://` vs `wss://`) | Confirm `ROX_RPC_HOST=0.0.0.0`, check the port (`9100` default), use `wss://` if TLS is on |
| "Invalid token" on CLI | Token mismatch between server and client | Re-export `ROX_SERVER_TOKEN` on both sides or regenerate with `openssl rand -hex 32` |
| Skill not triggering on `@mention` | Skill file invalid or workspace not active | Check `~/.rox/workspaces/<id>/skills/<name>/SKILL.md` exists and frontmatter parses |
| MCP server crashes on launch | Missing env var or unsupported Node/Python version | View MCP logs in Settings → Sources → click the source → "View logs" |
| Mission completion blocked | A gate is failing, or the final artifact is missing | Open the Mission Center → red gate → click "Why is this failing?" — the agent explains |
| Public share link returns 404 | Expired or revoked | Generate a new link from the session header |
| Agent gets stuck mid-tool | Tool timed out or hung | Click `Cancel` in the message header; the next turn picks up where it left off |
| Logs needed for a bug report | Debug mode | Launch with `-- --debug` and attach `~/.rox/logs/electron/main.log` |

For anything not in this table: ask the agent. *"Why is this happening?"*
usually works — ROX.ONE has access to its own logs and config.

## 11. Where data lives

ROX.ONE stores everything under `~/.rox/`:

```text
~/.rox/
├── config.json              # Workspaces list, LLM connections, app prefs
├── credentials.enc          # AES-256-GCM encrypted source credentials
├── preferences.json         # User preferences
├── theme.json               # App-level theme
├── logs/                    # Electron + server logs
└── workspaces/
    └── <workspace-id>/
        ├── config.json      # Workspace settings + RBAC scope
        ├── theme.json       # Workspace theme override (optional)
        ├── automations.json # Event-driven automations
        ├── sessions/        # JSONL session transcripts
        ├── sources/         # Connected sources + their state
        ├── skills/          # Custom skills
        └── statuses/        # Workspace-specific status definitions
```

### Migration shim from legacy `~/.craft/`

Upgrading from a pre-v1 build? On first launch v1.0.0 detects either
`~/.craft-agent/` or `~/.craft/` (preferring `~/.craft-agent/` if both
exist) and copies it to `~/.rox/`. This is a one-shot copy:

- Source is left intact for safety.
- If `~/.rox/` already exists, migration is **skipped** — the shim never
  overwrites.
- Logs of the copy land in `~/.rox/logs/electron/main.log` with the
  `[user-data-migration]` prefix.

See `docs/release/v1-migration-guide.md` for the full upgrade walkthrough.

## 12. Privacy and telemetry

ROX.ONE is opt-in for telemetry. Default install collects **nothing**.

### What we can collect when you opt in

- Crash dumps (no message content, no source content)
- Anonymized feature-use counts (which screens are opened, how often)
- Aggregate timing metrics

### What we never collect

- Your messages
- Your agent responses
- Your source content or credentials
- Your file paths or directory listings
- Your prompts or skills

### Opting in or out

Settings → Privacy → Telemetry. Toggling off removes the local opt-in flag
and stops the next launch from sending anything. Existing data on our side
is not retroactively deleted unless you ask for deletion explicitly via
`SECURITY.md`.
