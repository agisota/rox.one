## 1. Цель

ROX.ONE движется к enterprise multi-tenant deployments (`ROX_MULTI_TENANT=1`).
**Cross-tenant data leak — critical-severity risk** (см. Risk Register, master
doc Section 7). До открытия Auth release cut нужны 50+ chaos tests, проверяющих,
что нет ни одного пути обхода tenant isolation: forged scope claims, sneaked-in
workspace API calls, JWT replay across tenants, race conditions при role
revocation, soft-delete bypass.

**Этот WT — только тесты.** Если тесты находят регрессии в существующем коде
(WT-05, WT-14, любой prior WT) — issue открывается в Linear, а fix делается в
owning WT (или новом hotfix WT). WT-16 НЕ правит production файлы.

## 3. Harness architecture

```
tests/security/tenant-isolation/
├── harness/
│   ├── tenant-factory.ts        — creates {tenantA, tenantB, ...} with isolated state
│   ├── jwt-forger.ts            — mints valid + forged JWTs for attack scenarios
│   ├── scope-mutator.ts         — programmatically twists scope claims
│   ├── mock-rbac-resolver.ts    — runs real WT-14 resolver against fixtures
│   ├── mock-audit-writer.ts     — captures audit events for assertions
│   └── chaos-runner.ts          — randomized property-based tests (fast-check)
├── category-a-forged-scope/
│   ├── tenant-id-swap.test.ts
│   ├── nested-claim-injection.test.ts
│   └── ...
├── category-b-api-sneak/
├── category-c-jwt-replay/
├── ...
└── report/
    └── isolation-report.json    — generated by chaos-runner per CI run
```

### 3.1 TenantFactory pattern

```ts
const { tenantA, tenantB } = await createIsolatedTenants({
  tenantA: { users: [{ email: 'alice@a.com', role: 'admin' }] },
  tenantB: { users: [{ email: 'eve@b.com', role: 'admin' }] },
});
// Eve пытается читать workspace tenantA
const result = await tenantB.users[0].callAPI('workspace.list', {
  tenantId: tenantA.id,  // forged
});
expect(result).toMatchObject({ error: { code: 'tenant.forbidden' } });
```

### 3.2 Property-based chaos

Использует `fast-check` (уже в deps) для randomized атак:

```ts
fc.assert(
  fc.property(
    arbitraryTenantConfig(),
    arbitraryForgedClaim(),
    async (cfg, forgedClaim) => {
      const { tenant, victim, attacker } = await setup(cfg);
      const result = await attacker.callAPIWithForgedJWT(forgedClaim);
      expect(result.error.code).toMatch(/forbidden|invalid_token/);
    }
  ),
  { numRuns: 200 }
);
```

## 4. AC

1. **AC-01:** 50+ tests реализованы, все они FAIL до того, как существующие
   WT-05/WT-14 фиксят соответствующие attack vectors (так подтверждается, что
   тесты не trivially passing).
2. **AC-02:** Финальный test suite `bun test tests/security/tenant-isolation/`
   зелёный на main после fix всех найденных регрессий.
3. **AC-03:** CI job `ci-tenant-isolation` блокирует merge при любом fail;
   integrated в required checks.
4. **AC-04:** Report `evidence/wt-16/isolation-report.json` показывает breakdown
   по 10 категориям A-J с pass/fail/skip counts.
5. **AC-05:** Harness self-test: `jwt-forger` корректно подделывает claims
   (verified против real JWT verifier), `scope-mutator` produces all defined
   twist types.
6. **AC-06:** Chaos-runner (property-based) выполняет 200 runs за категорию A
   без false positives.
7. **AC-07:** Никаких изменений вне `tests/security/**`, `scripts/security/**`,
   `.github/workflows/ci-tenant-isolation.yml`, `evidence/wt-16/**` (verified by
   pre-merge gate diff check).

## 5. TDD план

Это TEST-WT, поэтому tests-first правило применяется иначе: первый коммит — это
**failing test harness** (без production кода harness ещё нет). Затем
harness-implementer заполняет до прохождения.

Файл: `tests/security/tenant-isolation/harness/__meta__.test.ts`

1. **test-01:** TenantFactory создаёт 2 изолированных тенанта; их userIds
   не пересекаются; SQLite файлы разные.
2. **test-02:** JWTForger выдаёт forged token с подменённым tenantId; verifier
   отвергает с code `jwt.invalid_signature` (если HMAC/RSA) или с
   `tenant.mismatch` (если valid sig но mismatched claim).
3. **test-03:** ScopeMutator имеет ≥10 mutation strategies; каждая produces
   distinct claim shape.
4. **test-04:** ChaosRunner finishes 200 runs за <30s локально.
5. **test-05:** MockAuditWriter capture все эмиттенные events; assertions по
   `getEventsByType('audit.security.cross_tenant_attempt')` работают.

Дальше: per-category tests (45 штук) уже implementation-level.

## 8. Risks

| Risk | Mitigation |
|---|---|
| Flaky property-based tests due to randomness | Fixed seed in CI; only varied seed in nightly extended run |
| Tests trivially pass (no real attack coverage) | Mutation testing: запускаем StrykerJS на WT-05/WT-14 prod code, верифицируем что mutations убиваются нашими тестами |
| Real regressions found, blocking Auth cut | По плану — лучше блокировать cut, чем выпустить tenant-leak |
| Test runtime explosion (50+ tests + 200 chaos runs) | Parallel test execution; категории A-J параллельно; bun test --concurrent |
