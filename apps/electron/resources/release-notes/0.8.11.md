# v0.8.11 — Сообщения, промпты, чат и исправления LLM

## Исправления
- **WhatsApp `selfChatMode` теперь симметрично фильтрует входящие** — `classifyInbound` ранее фильтровал только исходящий (`fromMe=true`) трафик, поэтому личное сообщение контакта или сообщение в группе всё равно маршрутизировалось в привязанную сессию даже при включённом режиме self-chat — агент мог отвечать от имени пользователя в не-self-chat переписке. Входящие из не-self-chat теперь отбрасываются при включённом `selfChatMode`, с новой причиной пропуска `non_self_chat_inbound` для ясности в логах. Обратная совместимость сохранена: при `selfChatMode=false` входящие от контактов обрабатываются как раньше.
- **`call_llm` восстанавливает частичный вывод вместо ошибки при достижении лимита SDK `maxTurns`** — значение `maxTurns` SDK в `queryLlm` было равно 1, поэтому вывод моделей рассуждений, естественным образом занимающий несколько SDK-витков (даже без инструментов), безусловно завершался ошибкой "Reached maximum number of turns (1)". Значение увеличено до 10, а потребитель теперь работает защитно: перехватывает частичный текст ассистента + предупреждение, когда SDK выдаёт error-result или выбрасывает исключение в середине потока, вместо возврата пустой ошибки. `call_llm` отображает `[Partial result — …]`, чтобы вызывающая сторона видела сигнал. Добавлены отладочные логи `[queryLlm]` (subtype, num_turns, stop_reason, errors) для устранения пробела в наблюдаемости.
- **Pi-бэкенд учитывает `includeCoAuthoredBy: false`** — Сессии Pi вызывали `getSystemPrompt()` без аргумента `includeCoAuthoredBy`, поэтому он молча принимал значение `true` по умолчанию и отображал блок Git Conventions даже когда пользователь отключил соавторство. Зеркальное отражение более раннего исправления, которое попало только на путь Claude. `getSystemPrompt` также получил защитный откат: если вызывающий опускает аргумент, он извлекает сохранённое значение `getCoAuthorPreference()` вместо захардкоженного `true`.
- **Цитата follow-up отправляется агенту целиком** — Большие цитаты follow-up-аннотаций молча обрезались примерно на 280 символах до попадания к агенту, поскольку настроенный по умолчанию лимит вспомогательной функции `normalizeExcerptForMessage` для тултипов применялся на пути к агенту. Путь к агенту теперь использует `normalizeFollowUpText` (только сворачивание пробелов, без ограничения длины); вспомогательная функция чипов переименована в `truncateForChipTooltip` и требует явного указания `maxLength`. Также удалена мёртвая 140-символьная предобрезка `noteLabel`.

## Улучшения
- **Вспомогательные функции follow-up извлечены и покрыты модульными тестами** — `formatFollowUpSection`, `truncateForChipTooltip` и `normalizeFollowUpsMarkdown` вынесены из `ChatDisplay.tsx` в соседний модуль `ChatDisplay.follow-ups.ts` с 13 целевыми тестами, покрывающими регрессию с цитатами более 1000 символов, каноническую форму сообщения, нумерацию множественных follow-up, парсер round-trip и поведение тултипов чипов. Добавлен узкий экспорт подпути `@rox-one/ui/annotations/follow-up-state`, чтобы тесты могли импортировать чистые функции без подтягивания pdfjs + Vite-специфичных импортов через barrel-файл UI.
- **Потребитель SDK-потока `queryLlm` извлечён для тестируемости** — Перенесён в `claude-llm-query.ts` с 12 новыми тестами, покрывающими пути выброшенных/возвращённых ошибок, структурированный вывод и блок рендеринга `call_llm`.
- **Pre-commit typecheck теперь автоматически обнаруживает TS-воркспейсы** — `scripts/typecheck-staged.sh` ранее содержал захардкоженный белый список из 8 воркспейсов, поэтому изменения TS в новых или переименованных воркспейсах (`messaging-*`, `apps/cli`, `apps/webui` и т.д.) молча пропускали хук. Теперь скрипт обходит `apps/<X>/` и `packages/<X>/` и проверяет типы в любом воркспейсе с `tsconfig.json`, предпочитая `bun run typecheck` при его наличии. Обнаружены и исправлены четыре ранее существовавших ошибки TS, которые скрывал старый белый список (типизация вебхуков в `apps/cli`, обновление lib до ES2023 + переименование `card.codeHtml` + guards indexed-access в `apps/marketing`, выравнивание tsconfig в `apps/webui` с Electron).
- **`electron:dev` теперь собирает WhatsApp-воркер при запуске** — `scripts/electron-dev.ts` собирал MCP-серверы и Pi agent server, но не WhatsApp-воркер, поэтому при свежих чекаутах возникала ошибка `MODULE_NOT_FOUND` на `dist/worker.cjs` при первой попытке подключения WhatsApp — это сбивало с толку, так как Electron-адаптер сообщал это как завершение воркера, а не как отсутствие сборки. Теперь dev-настройка вызывает канонический `scripts/build-wa-worker.ts` (~70 мс), поэтому бандл воркера остаётся синхронизированным при каждом запуске.
